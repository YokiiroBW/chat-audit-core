# 项目总任务队列

本文记录所有已完成项目、未完成项目和封存路线。后续推进时按编号顺序开发，每个可验收版本都需要测试、中文提交并推送到局域网 Forgejo；涉及运行时或部署行为变化时，还需要部署 NAS 并验收。

1. `FLOW-01` 明确本轮要推进的队列编号和验收目标。
2. `FLOW-02` 阅读相关代码、测试和历史文档，避免重复实现。

7. `FLOW-07` 使用中文提交信息提交。

## 3. 未完成开发队列

### 3.1 `Q-01` QQ 离线完整性验收

状态：第一轮代码完成，待生产样本持续验收。

前置：`DONE-05`、`DONE-11`。

目标：

1. 验证历史消息在断网后仍能完整浏览。
2. 覆盖文本、图片、动画表情、语音、视频、文件包/文档、卡片、合并转发、回复消息。
3. 对缺失资产给出明确原因，而不是只显示文件名或空白。

开发步骤：

1. 梳理 `/api/offline/audit` 对 QQ 消息类型的覆盖。
2. 增加缺失原因分类和测试样本。
3. 更新前端缺失状态展示。
4. 运行相关测试和必要的全量测试。

验收：

1. `/api/offline/audit` 没有高优先级缺失项，或全部有明确不可恢复原因。
2. 前端断网访问已缓存消息时，已入库媒体可预览或播放。
3. 缺失媒体显示可理解的缺失状态和修复入口。

## 4. 封存路线

### 4.1 `HOLD-01` 微信 PC / WeChatFerry 路线

状态：封存。

封存原因：

1. 新版微信 4.x 与当前 `wcferry 39.5.2.0` 不兼容。
2. 已安装并校验旧版微信 `3.9.12.51`，但本机 HVCI/内存完整性会阻止 WCF 注入链路。
3. 使用独立旧电脑或虚拟机会占用同一个个人微信号的 PC 登录，不符合当前使用约束。
4. PadLocal/iPad 协议和数据库读取方案均存在账号风险、外部依赖或媒体完整性不足问题。

保留内容：

1. `POST /api/wechat/events`
2. `POST /api/receive_external_msg`
3. `POST /api/external/media`
4. `POST /api/wechat/media`
5. `wechat_tray_adapter/`
6. 微信样本回放测试

恢复条件：

1. 用户明确重新开启微信路线。
2. 或出现稳定、低风险、不占用当前 PC 登录、可后台缓存媒体的可行方案。
